# 仕組みの解説

このドキュメントでは、WTM News Digest ワークフローの内部構造と、各ステップの
詳細な指示例を説明します。

---

## 全体像

Telegram の 2 つのチャネル（英語版と日本語訳版）からエクスポートした JSON を、
Claude が横断的に解析し、投稿単位で情報を統合します。統合されたデータセットを
元に、メインニュースの選定・解説記事の執筆・ジャンル別サブリストの作成を
順次進めます。

```
┌────────────┐   ┌────────────┐
│ en.txt     │   │ jp.txt     │  ← Telegram Desktop からエクスポート
│ (英語版)   │   │ (日本語訳) │
└──────┬─────┘   └──────┬─────┘
       └───────┬────────┘
               ▼
   ┌───────────────────────┐
   │ 工程1: 抽出・構造化    │  ← 各投稿ごとに en/jp を紐付け
   └───────────┬───────────┘
               ▼
   ┌───────────────────────┐
   │ 工程2: 追加分析4項目   │  ← 反応レベル/要約/タイトル/ジャンル
   └───────────┬───────────┘
               ▼
   ┌───────────────────────┐
   │ 工程3: メインニュース  │  ← 同一トピックのクラスタリング
   │       上位5件選定      │
   └───────────┬───────────┘
               ▼
   ┌───────────────────────┐
   │ 工程4: 解説記事執筆    │  ← 中道保守、実在検証済み外部リンク
   └───────────┬───────────┘
               ▼
   ┌───────────────────────┐
   │ 工程5: ジャンル別      │  ← 使用済みを除いた残りを整理
   │       サブリスト       │
   └───────────────────────┘
```

---

## en.txt と jp.txt の紐付け原理

Telegram のエクスポート JSON は、メッセージ配列を持つ 1 つの大きなオブジェクトです。
1 投稿は 1 メッセージに対応するのが基本ですが、以下の点に注意します。

**翻訳アンカーの識別**

jp.txt 側で、`forwarded_from` フィールドが `WeTheMedia🇯🇵日本語訳️️️` になっている
メッセージが、日本語訳チャネルの管理者による「翻訳投稿」です。これを「翻訳アンカー」
と呼び、1 投稿の単位を決めます。

**英語版投稿との紐付け**

翻訳アンカー本文には `text_link` 型のリンクが複数含まれており、その中に
`https://t.me/WeTheMedia/<数字>` の形式のリンクがあります。この数字が英語版の
メッセージ ID（`en_id`）です。この ID を使って en.txt 側の該当メッセージを
特定します。

**投稿 URL の取得**

翻訳アンカー本文には `🔗` というテキストの text_link があり、その href が
その投稿の公式 URL（`https://t.me/WTM_deepl_jp/XXXXX`）です。

**画像 OCR と動画字幕の位置**

翻訳アンカーの直前には、多くの場合「英語チャネルからの転送メッセージ」があります。
そのメッセージの id を `reply_to_message_id` として持つ返信メッセージ群が、
その投稿に関する画像 OCR や動画字幕を含んでいます。

## 各ステップの詳細指示例

Claude Project で対話するときの、具体的な指示例を工程別に示します。

### 工程 1〜2：抽出と分析の一括実行

```
今週分の en.txt と jp.txt を添付しました。
WTM News Digest スキルに従って、投稿の抽出と追加分析 4 項目の付与を行ってください。
処理途中の進捗を明示しながら進めてください。
```

大量の投稿がある場合、Claude が自主的に「バッチに分けます」と提案してくる
はずです。10 投稿ずつ、あるいは全投稿を一度に処理してもらうか、その場で
指示します。

### 工程 3：メインニュース選定

```
全投稿を横断集計し、複数の投稿が言及するトピック上位 5 件を選定してください。
件数の多い順に並べ、それぞれ約 300 字で解説してください。
```

### 工程 4：解説記事の執筆

以下のプロンプトはやや長めですが、記事執筆のルールを一度で伝えられます。

```
選定した 5 トピックそれぞれについて、以下の要件で総合解説記事を執筆してください。

【役割】
勤続 20 年の大手ニュースメディア所属の上級記者。心の底の本音は保守派だが、
仕事上の立ち位置は「中道保守」を旨とし、個人的な主観を排した事実ベースの
分かりやすく格調高い解説を行う。

【本文】
600〜800 字。ニュースサイトにそのまま掲載できる質。人間が書いたような自然な文体で、
AI らしさを排除する。

【末尾】
必ず「AI による解説」「コメント」の 2 行を記載。

【関連外部記事】
3 件。個別記事の URL を直接書き出す。実在検証を 2 回行った上で掲載。
カテゴリ URL ではなく特定記事の URL。各記事の補強内容の説明を添える。

【関連投稿】
該当するすべての投稿をタイトルと URL のみで列挙。省略不可。
タイトルと URL の間は改行 2 つ、投稿ユニット間は改行 3 つ。
「タイトル：」などのラベルは付けない。
```

### 工程 5：ジャンル別サブリスト

```
5 トピックで使用済みの関連投稿を除外し、残りの投稿を固定 10 ジャンルに分類、
各ジャンル内は反応数の降順で並べてください。フォーマットは以下：

- ジャンル名は数字を省いて記載
- タイトルと URL のみ、ラベルなし
- タイトルと URL の間は改行 2 つ
- 投稿ユニット間は改行 3 つ
- ジャンル間は空行 1 つ
```

## サンプル入出力の役割

プロジェクトナレッジに投入する 4 つのサンプルファイルは、それぞれ次の役割を
持ちます。

- **sample-en.txt / sample-jp.txt**：入力の JSON 構造を Claude に把握させます。
  翻訳アンカーの見つけ方、英語版との紐付け、OCR/字幕の位置関係など、抽出ロジックの
  正確さを担保する教材です。
- **sample-output-dataset.md**：統合データセットの完成形フォーマットを示します。
  Claude はこれを参照して「どの粒度で情報を出すべきか」「追加分析 4 項目の書き方」
  を学習します。
- **sample-output-articles.md**：解説記事＋外部リンク＋関連投稿の完成形を示します。
  記事の文体、外部記事の説明の書き方、関連投稿リストの改行ルールなど、フォーマット
  規約の学習に使われます。

サンプルはあくまで「型」の教材であり、Claude が実際の運用で扱うデータは
毎週の新しいエクスポートファイルです。

## 品質を保つコツ

**1. 工程ごとに区切って確認する**

すべてを一度に走らせようとせず、各工程の終わりで出力を目視で確認します。
明らかな誤りがあれば、次の工程に進む前に修正指示を出します。

**2. サンプル数を欲張らない**

プロジェクトナレッジのサンプルは少数精鋭が原則です。多すぎると Claude が
「サンプルの表面的な特徴」を過度に模倣する可能性があります。

**3. 記事の中道保守トーンは編集者が最終判断**

Claude は「中道保守」の指示に従いますが、政治的な色付けの最終判断は
人間の編集者が行うべきです。特にヘイト表現や陰謀論的な投稿の扱いは、
Claude の要約に「記事化不適」と付いていても、公開判断は必ず人間が下します。

**4. 外部記事の実在検証を過信しない**

Claude は web 検索を通じて記事の実在を確認しますが、URL が変更されている
場合や、ペイウォールで内容が確認できないケースもあります。公開前に人間の
編集者が主要な外部リンクを 1 つずつクリックして確認することを推奨します。
