# トラブルシュート — 既知の落とし穴と対処法

このドキュメントは、WTM News Digest ワークフローで実際に遭遇した問題と、
その対処法をまとめたものです。同じ問題に出会ったときの参考にしてください。

---

## en.txt が JSON パースエラーになる

**症状**：Claude が「en.txt を JSON として読み込めない」と報告する。

**原因**：Telegram Desktop からエクスポートした `result.json` を切り出す際、
末尾の閉じ括弧（`]\n}`）が欠けていることがあります。JSON パーサはこれを
不完全な JSON として拒否します。

**対処**：Claude に以下のように指示してください。

```
en.txt の末尾に "]\n}" を追記してから再度 JSON パースを試してください。
```

Claude Skill の `SKILL.md` にはこの処理が明記されているため、多くの場合は
Claude が自主的にこの手順を実行します。

---

## 付属メディアが投稿としてカウントされてしまう

**症状**：投稿数が予想より多く、内容の重複した投稿が並ぶ。

**原因**：1 つの投稿に複数のメディア（動画 + 画像など）が付いている場合、
Telegram エクスポート内では複数のメッセージとして分割されて記録されます。
このうち 2 つ目以降は「付属メディア」であり、独立した投稿ではありません。

**見分け方**：付属メディアはリアクションを持たないのが特徴です。en.txt 側で
`reactions` フィールドが空、または存在しないメッセージが該当します。

**対処**：Claude に以下のように指示します。

```
en.txt でリアクションを持たないメッセージ（付属メディア）を投稿数から除外して
再カウントしてください。
```

---

## 翻訳アンカーとのマッチングが失敗する

**症状**：一部の投稿で「英語原文が見つかりません」と表示される。

**原因**：jp.txt の翻訳アンカーの `forwarded_from` フィールドは
`WeTheMedia🇯🇵日本語訳️️️` という文字列ですが、末尾には見えない Unicode 文字
（variation selector など）が複数含まれています。文字列比較を部分一致で
行うと、意図しないメッセージがヒットする可能性があります。

**対処**：完全一致で比較する、または該当文字列をコピー元からそのまま渡す
ようにします。Claude に以下のように依頼できます。

```
翻訳アンカーの判定は forwarded_from の完全一致で行ってください。
基準となる文字列は jp.txt 内の実際のメッセージからコピーしてください。
```

## 反応レベルの数値が投稿間で不整合になる

**症状**：似たリアクション数の投稿なのに、反応レベル（1〜10）が大きく異なる。

**原因**：Claude は 1 バッチずつ処理する際、そのバッチ内での相対値で
レベルを判断する傾向があります。バッチをまたぐと基準がずれることがあります。

**対処**：投稿数が多い場合、全投稿の抽出が終わった段階で「最大リアクション数と
最小リアクション数を確認し、全体分布に基づいて反応レベルを再割り当てしてください」
と指示すると整合します。

## 画像 OCR が「無し」になってしまう投稿がある

**症状**：明らかに画像付きの投稿なのに、OCR 欄が「無し」と出力される。

**原因**：Telegram エクスポートで OCR/字幕データが取得できるのは、jp.txt 側で
「英語転送メッセージの id を `reply_to_message_id` として持つ返信メッセージ」に
情報が書き込まれている場合のみです。日本語訳チャネル側で OCR 処理が実施されて
いない投稿は、原理的に取得できません。

**対処**：これは仕様上の制約であり、「[データ不在]」と明記されるのが正しい動作です。
無理に推測で埋めないでください。

---

## 外部記事の実在検証が通らない

**症状**：Claude が「外部記事の実在確認ができませんでした」と報告する。

**原因**：一部の主流メディア（NPR、Washington Post など）は robots.txt で自動
クロールを制限しています。この場合、web_fetch では 403 エラーや ROBOTS_DISALLOWED
エラーが返ります。

**対処**：web_search で本文スニペットが取得できていれば、記事は実在すると
みなして構いません。Claude に以下のように指示します。

```
robots.txt でクロール制限のあるサイトは、web_search 結果に本文スニペットが
現れれば実在確認とみなしてください。web_fetch のエラーは無視して構いません。
```

## 記事の文体が「AI っぽく」なる

**症状**：解説記事に「〜と言えるでしょう」「〜が重要です」などの定型表現が
頻出する。

**原因**：中道保守の指示に忠実に従うあまり、Claude が過度に慎重・平板な
文体になることがあります。

**対処**：以下の指示を追加すると改善します。

```
次の表現を極力避けてください：
- 「〜と言えるでしょう」「〜が重要です」の多用
- 段落の冒頭に「まず」「次に」「そして」を並べる構成
- 「〜が求められています」「〜が期待されます」といった主体不明の受動態
- 結論部の「今後の展開が注目されます」

代わりに、能動態で主語を明確にし、事実の記述に徹してください。
比喩や引用は文の中盤に置くと自然です。
```

## Skills が自動起動しない

**症状**：ユーザーが「WTM のまとめを開始してください」と依頼しても、Claude が
`wtm-news-digest` スキルを読み込まない。

**原因**：Skills の自動起動は、SKILL.md の `description` フィールドとユーザーの
依頼文の類似度で判断されます。日本語でのマッチング精度は英語より若干劣ります。

**対処**：以下のいずれかを試してください。

- 依頼文に「WTM」「Telegram」「ニュースまとめ」などの具体的なキーワードを含める
- Claude に直接「`wtm-news-digest` スキルを使ってください」と明示する
- スキル管理画面で他のスキルを一時的にオフにし、競合を減らす

---

## Project のコンテキストウィンドウが逼迫する

**症状**：会話が長くなると、Claude の応答品質が明らかに落ちる。

**原因**：Claude Pro/Max の Projects は、RAG モードで知識ベースの一部だけを
コンテキストに取り込みますが、対話が長引くと会話履歴自体がウィンドウを圧迫します。

**対処**：以下のいずれかを試してください。

- 工程 4（記事執筆）の後で新しい会話を開始する
- 統合データセットを Markdown ファイルとして保存し、次回はそれをアップロード
  してから作業を再開する
- プロジェクトナレッジのサンプル数を最小限に絞る
